## Описание лабораторной работы
Лабораторная работа посвящена шаблонизации в PHP и разделению приложения на слои логики и представления.
За основу взят проект из лабораторной работы №6: форма добавления банковских транзакций с серверной валидацией, сохранением данных в `JSON` и сортировкой списка записей.

В рамках работы реализованы два варианта интерфейса:
- нативные PHP-шаблоны;
- шаблоны на Twig.

Также для дополнительного задания добавлен собственный Twig-фильтр `currency`, который форматирует сумму транзакции.

## Инструкции по запуску проекта
1. Проверить, что установлен PHP 8+:
```bash
php -v
```
2. Перейти в каталог лабораторной:
```bash
cd /Users/MAC/VSC/php_labs/lab07
```
3. Установить Twig через Composer:
```bash
composer install
```
4. Запустить встроенный сервер PHP:
```bash
php -S localhost:8000
```
5. Открыть страницы в браузере:
- `http://localhost:8000/index.php` — версия на нативных PHP-шаблонах;
- `http://localhost:8000/twig.php` — версия на Twig.

Если `composer` временно недоступен, страница `twig.php` покажет сообщение о необходимости установить зависимость `twig/twig`.

## Краткая документация к проекту
Структура проекта:
- `index.php` — точка входа для версии на нативных PHP-шаблонах.
- `twig.php` — точка входа для версии на Twig.
- `src/app.php` — общая логика и функции подготовки данных для обеих страниц.
- `src/Form/TransactionForm.php` — обработка `POST`, PRG, работа с ошибками и сохранением.
- `src/Storage/TransactionStorage.php` — чтение и запись данных в `JSON`.
- `src/Validation/*` — валидаторы полей формы.
- `src/Twig/TransactionTwigExtension.php` — регистрация пользовательского Twig-фильтра.
- `templates/php/*` — layout, форма и таблица для нативных PHP-шаблонов.
- `templates/twig/*` — базовый Twig-layout и частичные шаблоны.
- `data/transactions.json` — хранилище транзакций.
- `composer.json` — зависимость `twig/twig`.

Основные особенности:
- логика подготовки данных вынесена из шаблонов;
- PHP-шаблоны отвечают только за отображение;
- Twig-шаблоны используют `extends`, `block` и `include`;
- обе версии работают с одними и теми же данными;
- в проекте сохранена структура `src` из лабораторной работы №6, как разрешено в задании.

## Примеры использования проекта
Подготовка контекста страницы:
```php
$pageContext = buildTransactionPageContext(__DIR__ . '/data/transactions.json', 'index.php');
```

Рендер нативного PHP-шаблона:
```php
$content = renderPhpTemplate(__DIR__ . '/templates/php/page.php', $pageContext);
echo renderPhpTemplate(__DIR__ . '/templates/php/layout.php', [
    'title' => 'Lab 07 | PHP Templates',
    'activeView' => 'php',
    'content' => $content,
]);
```

Подключение Twig и рендер Twig-шаблона:
```php
$loader = new \Twig\Loader\FilesystemLoader(__DIR__ . '/templates/twig');
$twig = new \Twig\Environment($loader, ['autoescape' => 'html']);
$twig->addExtension(new TransactionTwigExtension());

echo $twig->render('index.twig', $pageContext);
```

Пример пользовательского фильтра:
```php
new TwigFilter('currency', static fn(int|float|string $amount, string $currency = 'MDL'): string => formatCurrency($amount, $currency))
```

Использование фильтра в Twig:
```twig
{{ transaction.amount|currency('MDL') }}
```

Примеры маршрутов:
- `http://localhost:8000/index.php`
- `http://localhost:8000/index.php?sort=amount&direction=desc`
- `http://localhost:8000/twig.php`

## Ответы на контрольные вопросы
1. В чём отличие нативных PHP-шаблонов от шаблонизатора Twig? Какие преимущества и недостатки у каждого подхода?

Нативные PHP-шаблоны используют обычный PHP-синтаксис прямо в представлении. Их преимущество в простоте и отсутствии внешних зависимостей, но при этом в шаблонах легче смешать бизнес-логику и HTML. Twig — это отдельный шаблонизатор с более строгим и безопасным синтаксисом, наследованием шаблонов, фильтрами и блоками. Он лучше подходит для поддержки крупных шаблонов, но требует установки и дополнительной настройки.

2. Зачем разделять логику и представление в проекте? Какие проблемы могут возникнуть, если смешивать их в одном файле?

Разделение упрощает поддержку, тестирование и повторное использование кода. Логика обработки данных, валидация и работа с хранилищем не зависят от HTML, а представление может меняться отдельно. Если смешивать всё в одном файле, код становится труднее читать, сложнее искать ошибки, а изменение интерфейса может случайно затронуть серверное поведение.

3. Что такое наследование шаблонов в Twig? Как работают `{% extends %}` и `{% block %}`?

Наследование шаблонов позволяет вынести общую разметку в базовый шаблон и переопределять только нужные участки. `{% extends %}` подключает родительский шаблон, а `{% block %}` объявляет именованный участок, который дочерний шаблон может заполнить или заменить. Это уменьшает дублирование и делает структуру представления более чистой.
